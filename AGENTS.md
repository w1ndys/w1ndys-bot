# Repository Guidelines

## 项目结构与模块组织

仓库目前处于初始化阶段，根目录仅包含 Apache 2.0 `LICENSE`，尚无源代码、测试或资源目录。引入实现时，请采用清晰且稳定的布局：应用代码放入 `src/`，测试放入 `tests/`，静态资源放入 `assets/`，脚本放入 `scripts/`，项目文档放入 `docs/`。测试目录应尽量映射源码结构，例如 `src/bot/handlers.py` 对应 `tests/bot/test_handlers.py`。不要提交构建产物、缓存、日志或本地密钥；新增工具链时同步维护 `.gitignore`。

## 构建、测试与本地开发

当前仓库尚未定义构建系统、依赖清单或可执行命令，因此不要假定 `npm test`、`pytest` 等命令可用。引入技术栈时，应提供单一、易发现的入口（优先使用 `Makefile` 或项目原生任务配置），并在 `README.md` 中记录环境要求。建议统一提供：

- `make setup`：安装或同步开发依赖。
- `make test`：运行完整测试套件。
- `make lint`：执行格式与静态检查。
- `make run`：启动本地开发实例。

提交相关改动时，请确保文档中的命令可以从仓库根目录直接执行。

## 编码风格与命名约定

遵循所选语言的官方风格，并提交对应的格式化、检查配置。使用 UTF-8、LF 换行和空格缩进，避免无关的全文件格式化。文件和模块名称应简短且表达用途；类型使用 `PascalCase`，函数与变量遵循语言惯例。配置项和环境变量使用 `UPPER_SNAKE_CASE`。公共接口应有简明注释，注释重点说明设计原因，而非重复代码行为。

## AI 辅助编程强制规范

所有由 AI 新增或修改的代码必须严格遵守以下规则：

- 每个文件顶部必须使用 `// 📌 影响范围` 列出该文件引用的全部外部变量；若无外部变量，也要明确标注“无”。
- 每个函数的函数头注释必须包含 `@param`、`@returns` 和 `⚠️副作用说明`。无参数、无返回值或无副作用时仍须显式说明。
- 函数内部每个 `if` 判断的正上方必须添加 `[决策理由]` 注释，解释判断存在的业务或安全原因。
- 每个函数最底部、函数结束前，必须以 `// >>> 数据演变示例` 开头，展示两组输入到输出的完整演变过程，包括关键中间状态。
- 所有 Git 提交信息必须使用中文描述，并在描述前添加一个符合改动含义的 emoji。统一格式为 `类型(内容): [emoji] 中文描述`，例如 `feat(ocr): 👏 新增功能`。

上述注释属于代码审查的必检项。AI 不得以逻辑简单、已有测试或语言惯例为由省略；所用语言不支持 `//` 时，应换用该语言等价的单行注释语法，同时保留标记文本。

## 测试指南

每项新功能和缺陷修复都应包含自动化测试，至少覆盖正常路径、边界条件和错误处理。测试应确定、隔离，不依赖真实外部服务；必要时使用 fixture 或 mock。测试名称应描述行为，例如 `test_rejects_expired_token`。目前没有既定覆盖率门槛；引入测试框架后，请在 CI 中执行并记录覆盖率基线。

## 提交与拉取请求

现有历史仅有 `Initial commit`。后续提交必须采用 `类型(内容): [emoji] 中文描述` 格式，例如 `feat(ocr): 👏 新增功能`、`fix(config): 🐛 修复配置读取失败问题`。类型使用 `feat`、`fix`、`docs`、`test`、`refactor`、`chore` 等 Conventional Commits 关键词；内容应简短说明影响范围。一次提交聚焦一个逻辑变更。拉取请求应说明目的、主要改动和验证命令，关联相关 issue；涉及界面时附截图，涉及配置或兼容性时明确迁移步骤。提交前确认测试、格式检查均通过，且没有密钥或个人配置进入版本控制。

## 安全与配置

密钥只通过环境变量或未跟踪的本地配置提供。新增配置时提交脱敏的示例文件（如 `.env.example`），并说明必填项、默认值及最小权限要求。依赖应锁定版本并定期检查已知漏洞。
